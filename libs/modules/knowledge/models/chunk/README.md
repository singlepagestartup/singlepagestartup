# Knowledge Chunk Model

## Purpose

Chunks are searchable fragments derived from source content. Each chunk stores text, parser metadata, and a 768-dimensional embedding for pgvector cosine search.

## Fields

- `id`: unique identifier (UUID).
- `createdAt`: creation timestamp.
- `updatedAt`: last update timestamp.
- `variant`: display variant.
- `text`: chunk text sent to search and generation context.
- `embedding`: `vector(768)` embedding generated by Ollama `nomic-embed-text`.
- `chunkIndex`: stable position within the source.
- `tokenEstimate`: approximate token count used to bound generation context.
- `contentHash`: hash of the chunk text.
- `metadata`: JSON metadata, including path, timestamp, heading, or parser details.

## Search Behavior

Custom knowledge search embeds the query through Ollama and orders chunks with pgvector cosine distance. The HNSW index is defined in the generated migration so search can use the database vector index.

Chunks connect to sources through the `sources-to-chunks` SPS relation instead of a direct `sourceId` column.

## Variants

- `default`: placeholder chunk view.
- `find`: data-fetch wrapper for querying chunks.
- `find-by-id`: fetch a chunk by ID on server or client.
- `admin-form`: admin create/edit form for chunk fields.
- `admin-select-input`: admin select input for choosing a chunk.
- `admin-table`: admin table listing chunks.
- `admin-table-row`: admin row showing chunk fields.

## Related API

Use `/api/knowledge/search` for vector retrieval and `/api/knowledge/generate` for grounded answers. Generated CRUD routes remain available through the chunk model SDK.

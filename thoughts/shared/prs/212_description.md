# PR Description Template

## Summary

Concurrent Telegram bootstrap requests could create duplicate RBAC grants and repeat free-subscription checkout work. This change makes both paths idempotent across processes with database-backed invariants, deterministic duplicate repair, and PostgreSQL advisory locking.

Closes #211.

## Changes

- Added startup-composable natural-key constraints for permissions, role permissions, and subject roles.
- Added transactional duplicate detection and repair, including fail-closed handling for conflicting data.
- Generated scoped Drizzle migrations and wired the repair step into aggregate repository migration.
- Added cross-process PostgreSQL advisory locking around free-subscription checkout with a second eligibility check inside the lock.
- Updated Telegram bootstrap so ordinary messages do not trigger checkout and concurrent topic/start events remain stable.
- Preserved request IDs in response-pipe errors and sanitized Telegram background-error logging and replies.
- Added unit and PostgreSQL integration coverage for uniqueness, repair, advisory locks, checkout concurrency, and Telegram concurrency.
- Documented startup overrides, natural keys, permission count, repair, and rollout procedure.

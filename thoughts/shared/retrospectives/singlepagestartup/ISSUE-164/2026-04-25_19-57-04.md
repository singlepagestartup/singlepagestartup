## Context

This session created and committed issue `#164` for porting the draft chat UI into the SPS subject social module. The workflow used the shared `core-00-create` path, GitHub project helper scripts, and the shared `commit` utility skill. Two workflow frictions were observed directly during execution: the initial sandboxed GitHub create sequence failed on network access and then continued into misleading downstream helper errors, and the commit flow predictably hit `.git/index.lock` permission failure before succeeding on elevated retry.

## Findings

### 1. GitHub create workflow does not fail fast after the first `gh` connectivity failure

- Surface: shared
- Type: command contract
- Impact: reliability
- Evidence: The first `bash -lc` issue creation block printed `error connecting to api.github.com`, then still ran `.claude/helpers/add_issue_to_project.sh` and `.claude/helpers/update_issue_status.sh` with empty `ISSUE_NUMBER` and `ISSUE_URL`, which produced secondary usage errors before the rerun with escalation succeeded.
- Root cause: The `core/00-create` contract requires a single-shell GitHub helper sequence, but the sample workflow does not require `set -euo pipefail` or an explicit guard that aborts immediately if `gh issue create` fails or returns an empty URL.
- Recommended fix: Refactor the shared `core/00-create` command to require a fail-fast shell block for GitHub create flows, or introduce a small shared helper that performs `gh issue create`, validates the returned URL/issue number, and only then runs project/status helpers.

### 2. Sandbox commit flow still pays a guaranteed failed attempt before elevated retry

- Surface: shared
- Type: automation opportunity
- Impact: speed
- Evidence: The first `git add`/`git commit` execution failed with `fatal: Unable to create '.git/index.lock': Operation not permitted`, after which the same write block was rerun with elevated permissions and succeeded.
- Root cause: The shared commit workflow documents the retry behavior, but execution still begins with a write attempt that is known to fail in this Codex sandbox for `.git` metadata updates.
- Recommended fix: Add a shared git-write wrapper or commit helper that detects this environment and requests elevated permissions before the first write operation, or centralize `git add` + `git commit` behind one approved path so the retry is not paid on every commit-oriented task.

## Quick Wins

- Update `.claude/commands/core/00-create.md` examples to include `set -euo pipefail` in the GitHub helper shell block.
- Add an explicit contract note that empty `ISSUE_URL` or `ISSUE_NUMBER` must abort the workflow before project/status helpers run.
- Add a small shared helper for commit writes so `git add` and `git commit` do not need an intentional first failure in sandboxed Codex sessions.

## Structural Improvements

- Consolidate GitHub issue creation into one shared helper used by both `.claude` commands and `.codex` skills so validation, fail-fast behavior, and escalation guidance live in one place.
- Move sandbox-aware git write behavior out of per-skill prose and into a reusable helper or wrapper command, reducing duplicated retry logic across future commit/publish workflows.

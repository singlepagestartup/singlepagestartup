## Context

Session context was the creation and scoping of issue `#162` for migrating `apps/host` to `next@16.2.4`. The workflow succeeded, but it exposed friction in the shared GitHub helper flow and in the `core-00-create` capture of migration-specific verification requirements.

## Findings

### 1. GitHub helper retries hide the need for immediate escalation

- Surface: shared
- Type: helper script
- Impact: speed, reliability
- Evidence:
  - The first `core-00-create` GitHub block failed repeatedly with `error connecting to api.github.com` after several `gh transient failure` backoff cycles before the same block was re-run with escalation.
  - `.codex/skills/core-00-create/SKILL.md` already tells the agent to rerun the unchanged `bash -lc` block with escalated network access when this error appears.
  - `.claude/helpers/gh_retry.sh` keeps retrying retryable API failures, but it does not surface a stronger signal that the caller should stop waiting and escalate in sandboxed environments.
- Root cause: The workflow knowledge about escalation lives in the command/skill contract, while the reusable helper only implements exponential retry. That split causes avoidable wait time every time sandboxed GitHub API access is blocked.
- Recommended fix: Add a shared helper or wrapper contract for GitHub sequences that detects `api.github.com` connectivity failures and emits a dedicated "rerun this exact block with escalation" exit path immediately after the first confirmed sandbox-style failure.

### 2. `core-00-create` has no dedicated slot for known regressions or mandatory verification gates

- Surface: shared
- Type: command contract
- Impact: reliability, maintainability
- Evidence:
  - Issue `#162` had to be amended after creation when the user added critical migration constraints: clean-state `build` and `start`, artifact cleanup before checks, and the prior OOM risk from issue `#113`.
  - The canonical template in `.claude/commands/core/00-create.md` captures title/problem/priority/size/type, but it does not ask for known regressions, risky validation paths, or mandatory verification steps.
  - Because of that gap, the ticket, process file, and GitHub issue all needed a second pass to carry the real acceptance risk.
- Root cause: The create-phase artifact structure is optimized for a generic issue description, but not for migrations or regressions where operator-held validation knowledge often arrives up front or shortly after.
- Recommended fix: Extend the create command/skill template with an optional section such as `Known Regressions / Required Verification`, and persist it into the ticket body plus GitHub issue body during initial creation.

### 3. The retrospective workflow assumes the artifact tree exists

- Surface: shared
- Type: artifact structure
- Impact: speed, portability
- Evidence:
  - Retro discovery produced `rg: thoughts/shared/retrospectives: No such file or directory (os error 2)` because the command searched the path before it existed.
  - The workflow could continue, but the missing path adds noise and forces ad hoc directory creation during execution.
- Root cause: The retrospective command specifies the output path, but the repository does not carry the base directory and the workflow does not normalize the path before scanning it.
- Recommended fix: Create `thoughts/shared/retrospectives/.gitkeep` in-repo or update the retro command wrapper to `mkdir -p` the base path before any search/read step.

## Quick Wins

- Add a tiny shared GitHub wrapper note or helper that returns a distinct failure mode for `error connecting to api.github.com` after the first retryable attempt.
- Extend `core/00-create` and `.codex/skills/core-00-create/SKILL.md` with an optional `Known Regressions / Required Verification` input section.
- Add the missing `thoughts/shared/retrospectives` tree to the repository so the workflow starts cleanly.

## Structural Improvements

- Centralize GitHub escalation behavior in reusable helper code instead of relying on each command/skill to remember when to stop retrying and rerun with escalation.
- Treat migration-specific validation requirements as first-class issue metadata across both `.claude` and `.codex`, so later research/plan phases inherit them automatically.
- Normalize artifact-path creation in workflow commands to reduce noisy shell failures and keep command outputs easier to scan.

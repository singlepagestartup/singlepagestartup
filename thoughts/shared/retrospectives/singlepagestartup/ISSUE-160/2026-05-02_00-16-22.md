## Context

Issue 160 implementation was committed and PR #168 was opened, but the GitHub Project item stayed in `In Dev` instead of moving to `Code Review`. The session also exposed helper compatibility drift and a missing default PR description template.

## Findings

### 1. Project helpers lost legacy function aliases

- Surface: shared
- Type: helper script
- Impact: reliability, portability
- Evidence: `.claude/helpers/get_issue_status.sh 160` failed because `load_config.sh` called `resolve_repo_context`, while `repo_context.sh` only defined `sps_resolve_repo_context`; project validation had the same mismatch for `validate_project_artifact_context`.
- Root cause: The repository context contract moved to `SPS_REPO_*` and `sps_*` functions, but older shared commands still called legacy names.
- Recommended fix: Keep `SPS_REPO_*` canonical while exporting `TARGET_REPO_*` and providing legacy wrapper functions in the shared helpers.

### 2. Utility PR flow bypassed implementation finalization

- Surface: shared
- Type: command contract
- Impact: reliability
- Evidence: `$commit` and `$describe-pr` succeeded, PR #168 was open, but issue #160 remained in `In Dev` because `core/30-implement.md` finalization step 9 was not executed.
- Root cause: PR creation and description are utility workflows, while the `Code Review` transition lived only in the full implementation command.
- Recommended fix: Add an idempotent finalization helper that verifies the PR, comments on the issue, updates status to `Code Review`, and can be called by `core-30-implement` or utility PR flows when they are used during implementation.

### 3. PR description workflow had no portable default template

- Surface: shared
- Type: artifact structure
- Impact: portability
- Evidence: `thoughts/shared/pr_description.md` was missing, so the PR description command had to fall back to an ad hoc body.
- Root cause: The command required a local template but the upstream project did not provide one for downstream projects to inherit.
- Recommended fix: Add a generic template under `thoughts/shared/pr_description.md` and let downstream projects customize it.
